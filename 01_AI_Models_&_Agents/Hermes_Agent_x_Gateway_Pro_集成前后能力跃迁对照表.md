# Hermes Agent × AI Agent Governance Gateway Pro 集成前后能力跃迁对照表

## 一句话结论

- 只做浅集成，不会发生真正质变，只会从强 Agent 运行时变成“带治理外观”的企业包装版。
- 做强治理闭环集成，才会发生质变：系统会从高权限个人 Agent Runtime 跃迁为企业级受控 Agent 执行平台。
- 该集成版不必强制采用 MIT，也不会因为集成了 hermes-agent 就自动变成必须开源的项目。

## 集成前后能力跃迁对照表

| 维度 | 集成前：Hermes Agent 单独运行 | 集成前：Gateway Pro 单独运行 | 集成后：强治理闭环版本 | 是否构成质变 |
| --- | --- | --- | --- | --- |
| 产品定位 | 高能力密度 Agent Runtime | AI 行为治理控制平面 | 企业级受控 Agent 执行平台 | 是 |
| 信任模型 | 默认单操作者可信 | 默认外层策略与审批优先 | 从“信任操作者”改为“信任控制平面” | 是 |
| 执行边界 | 默认可触达本地/远程高权限执行面 | 只治理动作，不自带强执行引擎 | 高风险执行必须经治理网关放行 | 是 |
| 审批模型 | 应用内审批与 break-glass 并存 | 外层审批、签名链、通知升级 | 审批权从应用内迁到外层平台 | 是 |
| 审计能力 | 有一定安全与审批意识 | 有完整 trace、audit、approval 证据链 | Agent 全生命周期可追溯 | 是 |
| 策略能力 | 以运行时约束为主 | 以策略、规则、审批、签名为主 | 工具调用和变更执行都变成策略对象 | 是 |
| 发布控制 | 无企业级灰度发布闭环 | 已具备 canary、progressive、full rollout 与 ingress 发布 | Agent 变更可纳入标准灰度发布面 | 是 |
| 供应链治理 | 有一定 CI 与供应链审计 | 具备外层准入与审批基础 | 插件、技能、安装器可进入企业白名单治理 | 部分 |
| 插件/技能生态 | 扩展性强但边界宽 | 可承载审批和策略编排 | 扩展安装从自由接入变成受控接入 | 是 |
| 多智能体协作 | 原生较强 | 原生不强 | 多智能体动作可纳入统一审批和 trace | 是 |
| 企业采购口径 | 更像研发/研究工具 | 更像治理平台或试点骨架 | 变成可被 CTO/CISO/运维联合采购的中台能力 | 是 |
| 直接准入能力 | 仍偏弱 | 有治理，但缺强执行引擎 | 取决于是否改掉 Hermes 默认高权限姿态 | 条件成立 |

## 哪些集成方式不会发生质变

### 1. UI 级浅集成

- 只是把 Hermes 会话、trace 或审批列表接到 Gateway Pro 页面里。
- 结果是“看起来更企业化”，但控制权仍在 Hermes 应用内部。
- 这类集成主要提升展示和可见性，不会改变系统本质。

### 2. 审计级旁路集成

- Hermes 继续自由执行，只在事后把日志、事件、trace 推送到 Gateway Pro。
- 结果是“事后可审计”，但不是“事前可阻断”。
- 这也不构成真正质变。

## 哪些集成方式会发生质变

### 1. 决策前置集成

- Hermes 的高风险工具调用先提交到 Gateway Pro。
- Gateway Pro 先做策略判定，再决定 allow、deny、approval_required。
- Hermes 不能绕开外层控制平面直接落地执行。

这会把系统从“Agent 自主执行”改造成“Agent 在制度内执行”。

### 2. 执行面托管集成

- 把 Hermes 默认 `terminal.backend: local` 改成受控后端。
- 例如容器化 runner、受控 SSH、受控远端执行器，而不是直接用操作者宿主机。
- Gateway Pro 负责发放执行凭据、限制作用域和记录回执。

这会把运行模型从个人工具变成企业执行基础设施。

### 3. 发布闭环集成

- 把插件发布、技能更新、规则集切换、Agent profile 变更都纳入 Gateway Pro 的 rollout 面。
- 低风险变更自动 canary，高风险变更必须 progressive/full approval。
- 发布结果进入 `rollout_executions`、`traffic_allocations`、trace search 与 ingress 发布面。

这一步会让 Agent 平台真正进入企业变更治理体系。

## 集成版最合理的产品形态

### 形态 A：受控引擎版

- Hermes Agent 作为执行引擎。
- Gateway Pro 作为治理、审批、审计、发布控制平面。
- 最适合企业试点、内网 PoC、受控自动化场景。

### 形态 B：企业封装版

- 对 Hermes 的本地高权限路径做删减。
- 默认禁用本地宿主机执行、自由插件安装、自由 break-glass。
- Gateway Pro 统一接管策略、审批、发布和审计。

这是最可能形成商业产品质变的版本。

## 关于许可证：是否必须 MIT

结论：**不是。**

### 你不必把整个集成版统一做成 MIT，原因有三点

1. Hermes Agent 是 MIT 许可证。MIT 是宽松许可证，不会强制衍生项目必须继续 MIT。
2. 你的 Gateway Pro 当前仓库并没有明确开源许可证文件，README 只写了“for demo, learning, and portfolio purposes”，见 [AI Agent Governance Gateway pro版代码/README.md](AI Agent Governance Gateway pro版代码/README.md)。这意味着它当前并不是一个标准意义上已授权的开源仓库。
3. 如果你是 Gateway Pro 的权利人，你可以对自己的新增代码、治理层、控制平面、商业交付层采用商业闭源许可、专有 EULA、双许可证，或者你自己选择的其他开源许可证。

## 关于开源：是否必须成为开源项目

结论：**也不是。**

### 三种常见情形

| 情形 | 是否必须开源 | 说明 |
| --- | --- | --- |
| 通过 API / sidecar / 独立服务方式集成 Hermes | 否 | 这是边界最清晰的方式，整体产品可保持闭源商业化 |
| 你分发一个打包版，里面包含 Hermes 原版或二进制 | 否 | 但必须保留 Hermes 对应的 MIT 版权与许可声明 |
| 你直接 fork 并修改 Hermes 源码后一起分发 | 否 | 仍不强制公开源码，但必须保留上游 MIT notice，不应把上游部分伪装成纯专有资产 |

## 真正要注意的不是“开不开源”，而是“怎么分层许可”

更稳妥的做法不是把整个仓库简单写成单一许可证，而是分层：

| 层 | 建议许可策略 |
| --- | --- |
| Hermes 上游代码及其修改部分 | 保留上游 MIT notice 与第三方声明 |
| Gateway Pro 核心治理控制平面 | 可保持专有商业许可 |
| 集成适配层 | 可选专有、MIT、Apache-2.0 或双许可证 |
| 文档、交付物、企业策略规则 | 可单独采用商业许可或内部使用条款 |

## 我对你这条产品线的实际建议

### 如果你要做商业集成版

推荐路线：

- Hermes 作为第三方 MIT 组件纳入。
- Gateway Pro 和企业治理封装层保持闭源商业许可。
- 对外附带第三方许可证清单和 NOTICE，而不是把整仓改成 MIT。

这是最符合你当前产品定位的做法。

### 如果你要做生态版或开发者版

可以考虑：

- 把集成适配器、SDK 或 connector 开源。
- 用 MIT 或 Apache-2.0 都可以。
- 如果考虑企业协作和专利边界，Apache-2.0 往往比 MIT 更稳。

### 如果你想做双轨产品

最现实的模型是：

- Community Connector：MIT 或 Apache-2.0
- Enterprise Governance Plane：专有商业版

这比把整个集成版一刀切成 MIT 更合理。

## 最终判断

- 做强集成版，能力层面会发生真实质变。
- 做集成版，不要求你必须 MIT。
- 做集成版，也不要求你必须开源。
- 真正需要做的是：把上游 MIT 组件、你的专有治理层、适配层和交付层的许可边界写清楚。
- 从商业与合规角度看，最稳的路径是“MIT 上游组件 + 专有企业治理层 + 明确第三方声明”。
